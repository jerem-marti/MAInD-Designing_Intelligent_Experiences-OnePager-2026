>
> **What this document is not.** An implementation plan. The implementation plan follows this spec; it lives in a separate file and is generated by the `writing-plans` skill after this spec is approved.

## 1. Front-door logic — two artifacts

`index.html` is the deliverable. `style-guide.html` is the appendix, linked from `index.html`'s footer as *"System documentation"*. This is the deliberate trade against the natural jury-confusion risk: the front-door page is the story, the linked document is the system proof.

```
ONE-PAGER/
├── index.html          ← NEW. Scrollytelling one-pager. The deliverable.
├── style-guide.html    ← V2 polish. Linked from index.html footer.
├── config.js           ← NEW. Single export: WEMATCH_AGENT_URL placeholder.
└── design/
    ├── tokens.css      ← Font collapse + new wash/badge tokens.
    ├── components/     ← review-card, match-card refreshed in place.
    └── sofia/          ← Existing 6 WebP stills. No new assets for V2.
```

## 2. Type — General Sans only

**Decision.** Drop Inter. General Sans carries display *and* body. Body weight 400, display weight 500–600.

**Why.** Two near-identical sans serifs read as indecision, not a system. General Sans has better editorial register at display sizes and is legible at 17px body. WeRoad's typographic family is closer to General Sans than Inter.

**Token change.**
```css
--font-display: 'General Sans', system-ui, sans-serif;
--font-body:    'General Sans', system-ui, sans-serif;
```
Drop the `<link>` to Inter on every HTML file.

## 3. Review card — tinted-paper variants

**Decision.** Each review card carries a low-saturation wash (green for positive, brick for critical) and a matching hairline border. Stars stay full-saturation. Sofia's §04 card uses the positive wash.

**Why.** At scan-speed the reader needs to see the polarisation before reading the words. Untinted cards on cream make the contrast typographic only — too quiet for the §02 job. The wash is calibrated low enough that the cream field still dominates; this is not a "branded with semantics" treatment.

**New tokens.**
```css
--color-positive-wash:  #EEF2EA;   /* paper + green-700 at ~6% */
--color-critical-wash:  #F4E8E5;   /* paper + brick-700 at ~6% */
--color-positive-edge:  #C5D2BC;
--color-critical-edge:  #DCC0B9;
```

**Component change.** `components/review-card.html` adds `background: var(--color-positive-wash)` to `.review--positive` and the critical counterpart. Border switches from `--color-border` to `--color-*-edge`.

## 4. Match card — editorial, detached from Figma

**Decision.** Detach from the Figma product card. The match card on this one-pager is a *designed artifact* whose job is to make the score and the reason readable in a glance — not to recreate WeRoad's commerce UI. Product-UI elements (Compare pill, heart, discount strikethrough, departure-date link, "See trip" button) are removed. The trip-card *idea* is implied; the page does not need to sell a trip.

**Why.** The §03b job is to communicate: *trips are now ranked by people-fit, and the system says why.* A literal product mockup turns §03b into "feature demo"; an editorial composition keeps it as a story beat. The Figma card stands as the production reality the team will ship — this page is the story above that reality, not a screenshot of it.

**Anatomy** (editorial):
- **Score** — the foregrounded artifact. Display weight, very large, in coral (preserves the single-accent rule). E.g. *87%*.
- **Reason line** — plain ink, directly below the score, micro letter-spacing wide. *"culture-driven · slow pace · mid-thirties mix"*. This is the WeMatch signature line the script v8 requires.
- **Trip name** — display medium, secondary visual weight. E.g. *Patagonia · 12 days*.
- **Optional thumbnail** — small, asymmetric, decorative. Designer call at implementation. May be omitted entirely.
- **WeRoad attribution** — micro, ink-400, e.g. *via WeRoad*. Quiet, just enough to anchor the artifact in the parent platform.

**No new vivid-green token needed.** Coral stays the single accent. The previously proposed `--color-match-badge` is retired before introduction — it would have broken the single-accent rule, and an editorial composition does not need a product badge.

**Layout in §03b.** Three cards in a horizontal row at desktop widths, stacked vertically at mobile. The 87% (top) card carries quiet emphasis — a coral hairline border or slightly larger score type — to mark it as the top match. The other two step down (79%, 71%) without ornament.

## 5. `index.html` — narrative spine

**Decision.** Six scroll segments. No horizontal section rules. Section number labels sit as small sticky chips in the left margin and fade in/out with scroll. A right-margin "live system" chip shows which token is dominant on screen — *the style guide as marginalia*. Sofia threads continuously from §02 to §04 inside a sticky container.

| Segment | Length (viewports) | Dominant element | Sofia state |
|---|---|---|---|
| §01 Hook | ~1.2 | Display-1 set low-left in empty cream field. "who" is the only coral. | hidden |
| §02 Truth | ~2.5 | Two tinted review cards + Sofia anchor + 18% stat (quiet large numeral) + closing line *"Group fit isn't measured. It's hoped for."* | hesitating |
| §03 Journey | ~5.0 | Reframe → entrance line → 3a (signals) → 3b (match cards re-rank) → 3c (the why panel) | curious → surprised → connected |
| Transition | ~1.0 | Single quiet pulse — Sofia on trip, one line | connected / settled blend |
| §04 After | ~1.5 | Sofia's review card in the same shape as §02 cards. Visual rhyme closes the loop. | settled |
| §05 Invitation | ~1.2 | "Meet WeMatch" CTA + micro-line + footer link to style-guide.html | settled, phone dark |

**Continuous, not folded.** Section dividers are removed. Rhythm is carried by type scale, whitespace bands, and Sofia's state transitions. The left-margin section chip (`§02 · TRUTH`) is the only navigational scaffolding visible during scroll. The right-margin "live system" chip is the editorial graphic-designer move: rather than catalogue tokens elsewhere, the page tells you in real time *which token is currently doing work* (e.g. `coral · used: 1`, `display-1 · 88px`, `--space-3xl`). Same content as a style-guide entry, but bonded to the moment it applies.

**Inside §03 — the three moments must read distinct.** Each moment is preceded by ~40vh of breathing room and signalled by a quiet number (`3a / 3b / 3c`) in the left margin. The §03b match-card row is the scan-mode lead — it's what a 10-second visitor will see first.

## 6. Sofia choreography — stills-first, VEO-ready

**Decision.** Phase A ships with crossfaded WebP stills synced to scroll. Phase B (when the team delivers VEO 3.1 clips) swaps each `<picture>` for a `<video>` and scroll progress drives `currentTime`. The Phase B path is structurally pre-wired and commented out — a one-block uncomment when the clips arrive.

**Mapping.**
```
SCROLL  0% ────── 20 ────── 40 ────── 60 ────── 80 ─── 100%
        §02              §03                       §04
       hesitating   curious   surprised   connected   settled
        [WebP]   ↔  [WebP] ↔  [WebP]    ↔  [WebP]   ↔  [WebP]
```

Sofia lives inside a sticky container spanning §02 → §04. Within that container, six `<picture>` elements are absolutely stacked. Each picture's opacity is driven by scroll progress through a single vanilla-JS handler: `IntersectionObserver` activates the container when in view, and an `rAF`-throttled scroll listener computes scroll progress within the segment and maps it to per-state opacity. No external library. ~40 lines of JS.

**Reduced motion.** `prefers-reduced-motion: reduce` → no crossfade; one still per segment, swapped on `IntersectionObserver`. No movement at all.

**Phase B swap path.** Each `<picture>` is wrapped in a comment block referencing the future `<video>` element. The scroll-scrub listener is written and disabled. The deliverable for Phase B from the production side: 5 VEO clips at ~2–3 s, VP9 in WebM, 720px tall, image-conditioned on the existing stills so the seam between clips matches.

## 7. CTA + live agent integration

**Decision.** A single static config file holds the agent URL. The CTA `<a>` element receives `href` from that config at load.

```js
// config.js
export const WEMATCH_AGENT_URL =
  "https://wematch-agent.placeholder.invalid"; // team replaces when live
```

```html
<!-- §05 -->
<a id="cta" class="cta" href="#">Meet WeMatch</a>
<p class="cta__micro">A live preview. Real agent. No signup.</p>
<script type="module">
  import { WEMATCH_AGENT_URL } from "./config.js";
  document.getElementById("cta").href = WEMATCH_AGENT_URL;
</script>
```

**Verb choice — "Meet WeMatch".** Lowest tech register of the three options in script v8. *Talk* implies a chat UI specifically; *Try* implies a product trial. *Meet* implies a person, which matches the agent's narrative role.

**Micro-line — verbatim from script v8.** *"A live preview. Real agent. No signup."* Eight words, three jobs (live / real / no friction).
